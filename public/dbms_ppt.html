THE SENSORY BACKBONE

Multi-Sensor Hardware Link

The system utilizes an ESP32 microcontroller as the primary data acquisition node, which interfaces with a localized sensor array. By polling raw analog and digital signals from high-precision units, the system creates a high-resolution map of environmental risks including Toxic Gas, Thermal Stress, and Spatial Obstruction.

These sensory vectors are formatted into sub-meter telemetry packets and streamed directly to the centralized command hub with sub-100ms latency, ensuring that the visual HUD remains a perfect digital twin of the physical factory environment.

THE DECISION BRAIN

Centralized Safety Logic

At the core of the system is the Safety Engine, a localized decision module that evaluates incoming telemetry against a multi-layered threshold matrix. This logic determines whether an environmental reading is nominal, a warning, or a critical emergency requiring immediate machine-level intervention.

When a breach is detected, the engine executes a bi-directional command: it broadcasts an alert to all administrative clients while simultaneously issuing a low-level hardware interrupt to the ESP32 to activate corrective actuators like Cooling Fans or Safety Buzzers.

DATA PERSISTENCE & INTEGRITY

Relational Trust Model

For an industrial system to be auditable, sensor data must be persisted with absolute integrity. Our system utilizes a high-performance MySQL backend with Active Connection Pooling, allowing it to handle sustained telemetry ingestion from multiple factory zones without overhead or data collisions.

To balance system longevity with real-time performance, we implement a Smart DB Throttle. While the UI displays every single packet at 50ms intervals, the database intentionally records at 2.5s intervals, ensuring a long-term historical record without exhausting disk I/O on unstable network hotspots.

THE DATABASE NEXUS (ERD)

9-Table Compliance Schema

The database architecture is designed with a strict emphasis on normalization and hardware-to-human traceability. Each machine instance is linked to a specific user and governed by a unique configuration of safety thresholds, ensuring Multi-Tenant Isolation across large-scale factory deployments.

THE VIRTUAL TWIN ENGINE

Simulation & Stress-Testing

The system includes a Stochastic Simulation Engine that generates high-fidelity synthetic telemetry. This "Digital Twin" replicates the natural drift of industrial sensors and allows developers to stress-test safety protocols under critical failure scenarios without risking physical hardware or personnel.

Every simulated packet is flagged within the database, allowing administrators to toggle between "Live Hardware" and "Simulation Mode" during analytics reviews. This ensures that training data and safety audits are never conflated with actual production telemetry.

BACKEND & SAFETY COMMAND

The Node.js Command Hub

Our server architecture utilizes an asynchronous event loop to handle concurrent connections from both factory employees and IoT hardware nodes. By leveraging the ws library, we maintain a persistent bi-directional nexus that allows for zero-refresh updates across the entire administrative dashboard.

Beyond data routing, the server implements active Zombie Pruning via a 30-second heartbeat check. This mechanism automatically terminates stale or broken connections, ensuring that the system's memory remains optimized for real-time safety critical tasks.

PERFORMANCE DATA STRATEGY

Hotspot Optimization (MySQL)

Industrial applications often operate in bandwidth-constrained environments. To combat this, we have optimized our database strategy to prioritize Network Stability over raw recording frequency. This approach prevents the connection pool from deadlocking during high-latency events on mobile hotspots.

By reusing a pre-allocated pool of 30 SQL connections and implementing optimized indexing on temporal columns, the system can retrieve a 24-hour visual history of thousands of records in under 150 milliseconds, providing instant operational context to decision makers.

EDGE HARDWARE PROTOCOL

ESP32 Command Node

The hardware layer is built on the ESP32's dual-core architecture, with firmware written in optimized C++. One core handles the high-frequency polling of the MQ2 and DS18B20 sensors, while the second core maintains the MQT/WSS Security Link back to the centralized control server.

This separation of concerns allows the device to respond to incoming fan/buzzer commands instantly, regardless of the network state. The inclusion of a 5V isolated relay ensures that the system can safely control industrial actuators without interfering with the sensitive micro-controller logic.
